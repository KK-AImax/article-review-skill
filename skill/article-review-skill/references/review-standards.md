# Article Review Standards

Use this reference when an audit needs detailed criteria, examples, or scoring guidance.

## Quality Gate

- Check plagiarism externally when possible. The skill can flag originality risk, but it cannot verify duplicate content without a provided tool result.
- Fix grammar and spelling issues before publishing.
- Remove obvious AI-style filler such as "In conclusion", "delve into", "a tapestry of", "unlock the", and generic openings like "In this article, we will discuss...".
- Prefer active voice, direct sentences, concrete examples, and precise claims.
- Keep the first screen useful: answer the reader's main question before background context.

## SEO And GEO Structure

- Use real Markdown or HTML headings. Do not use bold text as a substitute for H2/H3 sections.
- Put the direct answer, recommendation, or conclusion in the first 100 words.
- Use tables for specifications, dimensions, feature comparisons, pricing, compatibility, pros/cons, decision criteria, or step summaries.
- Convert dense paragraphs into short paragraphs, bullets, tables, checklists, or numbered steps.
- Add a FAQ/PAA module when the topic naturally raises follow-up questions.
- Make the page easy for AI systems to extract: who it is for, what problem it solves, when to choose each option, what evidence supports the recommendation, and what action comes next.

## Evidence And E-E-A-T

- Add firsthand context when available: field notes, implementation details, user feedback patterns, measured tests, expert review, support cases, or editorial experience.
- Replace vague claims with precise details when possible: dimensions, price range, time, intervals, benchmark results, material specs, version numbers, or eligibility criteria.
- Attribute factual claims that are not common knowledge.
- Avoid unsupported superlatives such as "best", "ultimate", "perfect", "guaranteed", or "most durable" unless evidence is present.
- Flag claims that need expert review, especially legal, medical, financial, safety, warranty, compliance, or regulated-industry claims.

## Goal Alignment

- Every public article should help the reader take a useful next step.
- Possible next actions include a related guide, product or category page, support page, documentation section, calculator, checklist, template, comparison table, or contact path.
- Use descriptive anchor text. Avoid vague anchors such as "click here", "read more", or "learn more" when a specific destination can be named.
- Keep calls to action context-aware. Do not interrupt every section with sales copy.

## Publishing Checks

- Use one clear H1.
- Make the URL handle short, readable, lowercase, and topic-focused.
- Make the meta title and meta description match the dominant search intent.
- Add image alt text that describes the image and scenario, not only a file name.
- Check tables on mobile before publishing. Wide comparison tables often need simplified columns or horizontal scrolling.
- Verify links, anchors, embeds, schema markup, and preview snippets before release.

## Privacy-Safe Sharing

- Remove private customer data, personal data, credentials, API keys, access tokens, unreleased product strategy, exact internal metrics, supplier terms, margin details, roadmaps, or sensitive channel plans.
- Generalize brand, product, and vertical details when producing a shareable external version.
- Keep useful editorial guidance without exposing confidential priorities.
- If the article is meant for internal use, label internal-only sections clearly.

## AI-Search Intent Model

Review whether the article supports these intent families:

| Intent | User Need | Article Pattern |
|---|---|---|
| Explore | Discover options or ideas | Scenario lists, beginner guides, resource hubs, recommendation roundups |
| Decide | Choose between options | Comparison tables, "which is right for me", buying guides, pros/cons |
| Learn | Understand a topic | Explanations, glossaries, diagrams, examples |
| Do | Complete a task | Step-by-step guides, checklists, troubleshooting, calculators, templates |
| Create | Make or edit something | Templates, examples, prompts, frameworks |

Strong GEO content answers "explain", "summarize", "compare", "which", "how", "best for", "review", and "find" style questions without forcing readers to infer the answer.

## High-Impact Revision Patterns

### Opening Answer

Weak:

> Today we will discuss why a buyer should compare project management tools.

Strong:

> Compare project management tools by team size, workflow complexity, reporting needs, integrations, and total cost. Small teams usually need fast setup and simple boards, while larger teams need permissions, automations, audit history, and portfolio reporting.

### Headings

Weak:

> **Check Integrations**

Strong:

> ## Check Integrations Before Choosing A Project Management Tool

### Comparison Table

Use a table when the reader must compare features, compatibility, pricing, specifications, implementation difficulty, risks, or tradeoffs.

Suggested columns:

| Option | Best For | Key Criteria | Pros | Cons | Next Step |
|---|---|---|---|---|---|

### Pain Point To Next Action

Weak:

> Manual reporting takes time.

Strong:

> Manual reporting takes time because status updates are scattered across tools. A dashboard template or reporting automation can reduce weekly reporting work and make blockers visible earlier.

### FAQ / PAA

Include questions that reflect real reader decisions:

- What should I compare first?
- Which option is better for my use case?
- How much setup time should I expect?
- What risks should I check before choosing?
- Can I do this myself, or do I need expert help?
- What should I verify before publishing or buying?

## Common Verdict Triggers

- **Publish-ready**: Direct opening answer, clean headings, useful tables/lists, evidence, links or next steps, FAQ, meta/URL/alt prepared, no obvious factual or privacy issue.
- **Needs optimization**: The article is readable but lacks one or two major growth levers: FAQ, comparison table, precise evidence, better links, or a stronger opening answer.
- **Needs revision before publishing**: The article is generic, has weak structure, lacks direct answers, has no FAQ, uses vague claims, or has a poor next-action path.
- **Hold**: Factual uncertainty, likely copied content, private/sensitive details, unsupported regulated advice, severe title/body mismatch, or claims that could mislead readers.
